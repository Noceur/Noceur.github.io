Digital Logbook Inception

Welcome. I am a Technical Level Designer deeply focused on optimizing content generation workflows and building custom programmatic toolsets for virtual world building.

My absolute priority is designing tools and levels that empower teams to scale content. Having spent years at studios like Sandbox Interactive (Albion Online), Stratosphere Games (Homeworld Mobile), and Spectarium Games (Myths), I’ve directly experienced where pipelines choke.

My philosophy revolves around architectural honesty — if code or structural patterns are redundant, build a script to automate it. This site is created to serve as an interactive record of my professional levels, custom procedural scripts, Houdini HDAs, and Maya batch exporters.

Skills & Technologies

  • Level Editing: Unity, Unreal Engine 5, Tiled Editor, Graph Tooling.
  • Procedural Systems: Houdini Engine (HDAs), Flip Book fluids, Vellum mesh wrappings.
  • Scripting Languages: Python, C# Scripting, Maya commands API, Mel, JSON schemas.
  • Asset Creation: Maya rigging/modeling, ZBrush sculpting, Quixel, Substance Suite.

Core Commitments

  • Reducing level build overhead from 30 minutes to <60s using programmatic Excel and Node setups.
  • Eradicating designer wrist strain and carpal tunnel via macro paddings and automation.
  • Preserving pristine collision & collision mesh hierarchies in multiplayer layouts.
  • Sharing free open source Python scripts, solvers, and technical breakdown guides.

Spectarium Games

Myths

Sole Level Designer · Unity · 3 biomes · ~150 zones shipped

A live-service mobile ARPG built around 4–6 minute dungeons. I owned the level-design pipeline end-to-end across three biomes — the zones, the tools that built them, and the procedural-content roadmap that was going to scale it.

Level Design

Every zone went through the same pipeline: blockout → tile layout → encounters → decoration → custom navmesh → playtest. Each zone shipped with a visual mesh, a walkable mesh for the client, and a navmesh shared by client and server.

Unity ~150 zones authored 3 biomes shipped Sole LD
Zones — three biomes

Norse — 5×5 to 20×20 outdoor tiles with vertex-painted shadows because baked light was out (runtime stitching broke seams) and dynamic light was out (mobile budget). Four rotation variants per tile, auto-swapped by a custom editor tool.

Mayan — 2×2 interior tiles with worldspace-textured floors and camera-aware walls split into a top piece and a stub bottom piece, so the camera could fade the top out without losing the floor plan.

Watcher — brighter interior that ditched worldspace texturing for deliberate per-tile patterns and trim seams. Carried the FTUE.

Dungeons — scriptable assembly

A dungeon was a sequence of zones — starter, regular/scenario zones in the middle, exit at the end, with small adapter zones bridging mismatched borders. Hand-assembled inside a scriptable object via a visual tool that read prefab positions relative to the starter zone and emitted the SO directly. Weather and biome meta-fields rode along.

Procedural strategy

The PCG roadmap split decoration into three layers with different control granularity: variants (whole zone chunks chosen per theme), compositions (size-standardised sub-pieces — S/M/L/XL corners, walls, freestanding) and micro details (per-run swappable props that never touch colliders).

On top of that, a dungeon graph defined zone constraints — theme list, encounter density, optional vs mandatory, fixed vs varying — and randomised order within those rules. That’s where partial control lives: force two scenarios into connected zones, lock a starting biome, guarantee a miniboss before an exit.

Editor Tools & Macros

Small scripts wired one task each — snapping, rotating, swapping tiles for their lighting-variant siblings, generating meta-data scaffolds for new zones. With them running I could lay out a 100×100m zone in around five minutes.

For raw efficiency I worked off a layered macro pad mapping the most-used tools by rotation in a numpad shape. Strongly recommended if you would like to keep your wrists.

Result & Reflection

Shipped: ~150 zones across 3 biomes, 100×100m zone authoring in ~5 minutes, two of five planned biomes completed end-to-end with their bosses, FTUE on the Watcher biome, PCG roadmap fully documented and ready to execute.

In hindsight I’d push back harder on the constraints that drove the vertex-painted-shadow system, and start the dungeon-graph work earlier so the first biome ships on top of it rather than retrofitting later.

Sandbox Interactive

Albion Online

Level Pipeline & Tools · Unity · hundreds of 1×1km maps

A single-world MMO where every map was hand-built — until a 10× playerbase made that stop scaling. I worked on the template-based level system and the tools that let one designer assemble a level in under a minute instead of half an hour.

Level Design & Grid Automation

Two systems took the brunt of the rebuild: templates and layers. Templates let levels be assembled out of pre-built chunks instead of from a blank scene; layers let those chunks vary across instances without re-authoring them.

The main template defined the silhouette and the slots for everything else. Border templates defined exit configurations per side. Content templates filled the slots — monster camps, castles, treasures, dungeon entrances, resource hotspots. A road system guided players between exits, built as layers connected by 4×4m crossing templates so any two chosen exits could be assembled cleanly.

Sheets-driven assembler

Templates solved authoring; they didn’t solve assembly. Stitching still meant moving thousands of objects in Unity and the editor wasn’t happy doing that.

The tool I built side-stepped the editor entirely. It pulled the location of every slot in a main template, the centerpoint of every available content template, and dumped the whole thing into a Google Sheet — one tab for data, one as the assembler UI. Designers got dropdowns per slot with thumbnails, exit pickers that fed Dijkstra to compute the road-layer combinations, and an export button that emitted level JSON for Unity to consume.

Asset Manager Renaming Tool

Every environmental prop had nine-plus variants (tier × biome) plus meta files. Renaming one asset meant touching eighteen files across nine folders. Artists hated it.

The asset mover let the user select any one file and type the new name. The tool returned a preview of suggested actions across every related variant, the user accepted or rejected, and a log file recorded the final operations. The bulk of the work was a regex layer that knew which parts of each path were substitutable and which were load-bearing.

Result & Reflection

Level assembly dropped from 15–30 minutes to under a minute, and was substantially less error-prone since templates and layers carried the constraints rather than relying on the designer to remember them. The asset mover removed an entire class of rename mistakes from the day-to-day workflow.

The Sheets layer was deliberately temporary — fastest path to a working tool, and it let non-engineers iterate on the assembly experience without code reviews. Long term, the natural next step was a native Unity editor window with the same UX.

Stratosphere Games

Homeworld Mobile & Technical Arts

Sole LD on Homeworld Mobile · minor on The Desolation · Python tooling on Dawn of Ages

Sole-LD work on the Homeworld Mobile tutorial and narrative missions, a brief stint on The Desolation’s early levels, and a Python pipeline tool for the Dawn of Ages animator who was tired of repeating the same export steps by hand.

Homeworld Mobile — tutorial & narrative

A mobile entry in the Homeworld universe needed a tutorial path and a cinematic narrative arc that played cleanly off the standalone systems for combat, progression and unlocks. I was the sole level designer on the project and focused on the tutorial and narrative-mission chain.

All mission logic was authored in an in-house node-based tool similar in spirit to Unreal Blueprints — nodes for triggers, fleet spawns, camera moves, narrative beats, and unlock gates. The unlock-gate nodes wired missions directly into the global progression graph.

Dawn of Ages — Maya batch rig exporter

An animator was burning through hours repeating the same FBX export setup for every rig change: strip the control rig, strip specific bones, push the studio’s FBX settings, export, then put the user’s settings back — multiplied by every rig that needed re-exporting that day.

I built a Maya tool in Python with JSON-defined profiles. Each profile stores its own “objects to keep,” “objects to delete” and target path, so a single button runs the full sequence per rig, or all of them in batch. Every export is wrapped in an undo chunk; the user’s personal FBX settings are pushed/popped around the studio-forced ones so nothing in their session is mutated when the run finishes.
