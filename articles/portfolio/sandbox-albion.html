Albion Online

Level Pipeline & Tools · Unity · hundreds of 1×1km maps

Albion Online is a PC first cross-platform sandbox MMORPG featuring a completely player-driven economy and a classless, gear-based combat system. The game requires highly scalable environments to support both intricate PvE dungeon encounters and massive, full-loot guild wars over territorial control across distinct, resource-driven biomes.

Level Design — the problem and the approach

Problem

Albion Online takes place in a single un-instanced world made of hundreds of 1×1km hand-built maps stitched together by loading zones. Great for immersion. Not so great when the playerbase grows tenfold and every prop on every map was placed by hand.

The old workflow was a designer painting a level from scratch in Unity, asset by asset. Throughput was bounded by headcount, every map was a one-off or repeated which players noticed.

Old monolithic level setup — everything authored in place.

Approach

Two systems took the brunt of the rebuild: templates and layers. Templates let levels be assembled out of pre-built chunks instead of from a blank scene; layers let those chunks vary across instances without re-authoring them.

The template stack

The main template was the glue. It defined the silhouette of the level and carried slots for everything else — it didn’t hold a lot of gameplay itself but baked in layers for variation and the optional roads underneath.

Main template — the structural backbone of the level.

Border templates defined where exits lived on each side of the map. They also used the layer system so the same border could host different exit configurations.

Border templates — layered exit configurations.

The holes in the main template were filled by content templates. This is where the majority of the game actually took place:

  • Monster camps
  • Castles & guild territories
  • Treasures
  • Dungeon entrances
  • Resource hotspots
Content templates — the gameplay-bearing chunks.

On top of all of that, a road system guided players between exits. Roads were layers within the main template, connected by tiny 4×4m crossing templates so the path between any two chosen exits could be assembled cleanly.

The remaining problem — and the Sheets-driven assembler

Templates and layers solved the authoring problem, but they didn’t solve the assembly problem. Stitching templates together still meant moving thousands of objects around in Unity at once, and the editor wasn’t happy doing that.

The tool I built side-stepped the editor entirely. It pulled the location of every slot in a main template, the centerpoint of every available content template, and dumped the whole thing into a Google Sheet — one tab for the data, one tab as the assembler UI.

Inside that sheet, the level designer had:

  • Dropdowns per slot, each showing a thumbnail preview of the template.
  • Exit pickers that fed into Dijkstra to compute the road-layer combinations that connect the selected exits.
  • An export button that emitted a level JSON for the Unity project to consume.
Complete templated level — assembled, exported, ready to load.
Asset Mover — the smaller tool that paid off daily

A neighbouring problem: each environmental prop had nine-plus variants (tier × biome) plus their meta files. Renaming or moving one asset meant walking nine folders and touching eighteen files. Artists and designers hated it.

The asset mover let the user select any one of those files and type the new name. The tool returned a preview of the suggested actions across every related variant, the user accepted or rejected, and a log file recorded the final operations. The bulk of the work was a layer of regex that knew which parts of each path were the substitutable bits and which were the load-bearing structural bits.

Result & Reflection

Result

Level assembly dropped from 15–30 minutes to under a minute, and was substantially less error-prone since the templates and layers carried the constraints rather than relying on the designer to remember them. The asset mover removed an entire class of rename mistakes from the day-to-day art workflow.

Reflection

The Sheets layer was deliberately temporary — it was the fastest path to a working tool, and it let non-engineers iterate on the assembly experience without code reviews. Long term the natural next step was to fold the same dropdown-and-preview UI into a native Unity editor window, drop the export round-trip, and lean harder into auto-generated road layers when more exits opened up.
